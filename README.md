# OPC 超级个体橙皮书

> **One-Person Corporation — 单人公司的系统架构、工程实践与组织防御**
>
> 2026 年 7 月 · 第 1 版

---

## 这是什么

本书讨论一个正在发生的组织范式迁移：**在 AI 执行能力、工具调用能力与外部服务 API 化程度持续提高的条件下，部分知识型业务的最小有效组织规模，正从「多边人际协同团队」收缩为「单个高认知个体 + 硅基自动化执行网络」**。

我们把这种形态称为 **OPC（One-Person Corporation，单一人格超级组织）**。

全书定位不是商业鸡汤，而是一份**系统架构与组织防御手册**——核心问题是：如何把一个高认知的个人，重构为一套具备自愈能力、鲁棒性极强的自动化商业实体。

---

## 核心工程假设与铁律

- **工程假设**：当 AI 能执行意图、能调用工具、外部服务高度 API 化时，单人 + 自动化网络在大量知识型业务上可等价于一个小团队。
- **全书铁律**：低风险动作自动执行（L3/L4），中风险动作半自动（L2），高风险动作必须人类物理确认（L0/L1）。

### L0 ~ L5 自动化分级协议

| 等级 | 含义 | 典型工程表现 |
|------|------|--------------|
| L0 | 人工执行 | 签署股权文件、高价值谈判 |
| L1 | AI 辅助 | AI 写开发信，人审核并发送 |
| L2 | 半自动 | AI 生成合同，进入待签核队列 |
| L3 | 条件自动 | 预设规则内自主执行，超标熔断 |
| L4 | 高自治 | 全链路自监控，异常自动回滚 |
| L5 | 完全自治 | 严禁生产环境，仅限闭环模拟实验 |

## 反复出现的核心工程原则

- **Verifier 不可省略**：Critic-Generator-Verifier 三体架构中，只有 Verifier（确定性代码 / Pydantic 校验 / L2 人工）拥有通行判定权，是系统的收敛内核。
- **星型拓扑消灭死锁**：多 Agent 系统用 Supervisor 星型拓扑，子 Agent 之间无通信边，从根上消除有向环死锁；单个 Supervisor 管控的子 Agent 不宜超过 4 个（3 个时系统可靠度 ≈ 0.857，15 个时骤降至 ≈ 0.463）。
- **幂等过滤**：事件总线中消息重复消费概率近 100%，涉及财务变动必须基于唯一事务 ID 做幂等过滤。
- **三层防线执行安全**：Pydantic 强类型校验 + 微服务临时 Token 隔离 + 工具白名单静态映射；代码在 Docker 断网沙箱中运行，超时强杀。
- **财务熔断**：Redis 滑动窗口，5 分钟内 API 消耗突破阈值自动拉闸并告警。
- **冷热数据三级隔离**：热（Redis，30min TTL 自毁）/ 温（PostgreSQL 只读网关）/ 冷（本地加密 SQLite，Append-Only）。
- **记忆隔离红线**：向量检索必须强制挂载用户 ID 过滤，且过滤发生在物理数据库层；绝不让大模型决定是否过滤——跨租户记忆混淆是最致命的反模式。
- **创始人冗余（SPOF）**：创始人是系统唯一单点故障；系统应支持 HRV / 离线联动降级到 L2 保守模式，并严守生理三红线（睡眠 < 6h 禁深度工作、每 90min 强制切换、16:00 后创作关闭）。
